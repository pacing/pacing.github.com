Oct 16th, 2012 | Comments

这一篇对Git对象进行总结。

首先看一个例子。

{% codeblock %}

git log -1 –pretty=raw

commit 7213e9b40359e477080ed8feeb1120041cbd5ee8 tree de4ca4100e63695044400bf7e2b834d6b053a62f parent 60c5a8067cadc872391a9890f89827264ca39015 author pacing wqy874654098@gmail.com 1350305952 +0800 committer pacing wqy874654098@gmail.com 1350305952 +0800

modify bye

{% endcodeblock %} 可以看出来，在一次提交中，包含了三个SHA1哈希值来表示对象ID。

再来一段代码。

{% codeblock %}

git log -2 –pretty=raw

commit 7213e9b40359e477080ed8feeb1120041cbd5ee8 tree de4ca4100e63695044400bf7e2b834d6b053a62f parent 60c5a8067cadc872391a9890f89827264ca39015 author pacing wqy874654098@gmail.com 1350305952 +0800 committer pacing wqy874654098@gmail.com 1350305952 +0800

modify bye

commit 60c5a8067cadc872391a9890f89827264ca39015 tree c722a9a8a289e99fcf2be5f658a67c8d7d94e8f7 author pacing wqy874654098@gmail.com 1350305791 +0800 committer pacing wqy874654098@gmail.com 1350305791 +0800

first commit

{% endcodeblock %} 对比可以发现

commit ***ID值****    对应了本次提交的唯一标识

parent ***ID值****    对应了本次提交的父提交（上一次提交的commit的ID值）

而 tree *ID值** 对应了本次提交所对应的目录树

git cat-file 这个命令可以来研究git对象ID。 {% codeblock %} git cat-file -p ID值 #这个命令可以显示对象的内容 git cat-file -t ID值 #这个命令可以显示对象的类型 {% endblock %} 由上面的例子，我们可以猜想到git版本库的大致结构。

前一次提交的commit和后一次提交的parent的对象ID相同，所有相邻两次提交都是如此，那么就构成了一条历史追踪链，也使得版本管理成为可能。

而每一个commit对象ID都对应了一个tree对象ID，对象ID又对个了自己的内容ID值。

至此，我们在原理上相同了git版本库的工作流程，当然，实现很复杂。

我们来慢慢解读。
